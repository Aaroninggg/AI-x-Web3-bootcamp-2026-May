## 一、AI 核心概念

### 1.1 LLM 大语言模型

> **个人总结：LLM 是"概率猜口"，不是"数据库"**

| 核心认知 | 原文摘要 |
|----------|----------|
| LLM 不能获取外界最新信息 | 单 LLM 本身不是数据库 |
| LLM 输出的是概率合理的候选，不是事实 | 生成的是概率上合理的输出，不是天然可信的 |
| 模型负责推理，人类负责定稿 | 模型能力是推理入口，不是最终验证 |
| 把输出变成可检查对象（schema/日志/引用） | 不能只停留在自然语言 |

**💡 你的洞察：**
> "AI 只能开脑洞，不能下定论"

### 1.2 Token & Embedding

| 概念 | 一句话 |
|------|--------|
| **Token** | 文字被切分后的最小单位，模型处理的"词" |
| **Embedding** | 把文字变成数字坐标，意思越像距离越近 |
| **本质** | 浮点数列表（向量），如 `[-0.0069, -0.0053…]` |
| **能干什么** | 看相似度，不能判断正确性 |

**🔗 主要用途（7 大类）：** 语义搜索、文本聚类、内容推荐、异常检测、多样性分析、文本分类、代码搜索

### 1.3 Transformer & Attention

| 核心认知 | 说明 |
|----------|------|
| Transformer = LLM 的骨架 | 当前主流 LLM 使用的底层结构 |
| Attention = 自动看前文中重要的内容 | 找规律、拼逻辑、串联上下文 |
| 优点 | 能串通所有人类提供的资料，通顺表达 |
| 缺点 | 没有自己的知识库，信息缺失就瞎编 |

### 1.4 Hallucination 幻觉

| 本质 | 看着通顺合理，实际全是假的 |
|------|-----------------------------|
| **风险** | 工作中用了错误信息会出大问题 |

**5 个解决方案：**
1. 强制标注资料来源
2. 用固定格式严格校验输出
3. 定死硬性规则核对
4. 重要内容人工复核
5. 全程留记录方便查错

### 1.5 Multimodal 多模态

- LLM 不只处理文字，多模态 AI 能看文字/图片/声音/视频/截图
- 优点：覆盖日常办公多模态场景
- 缺点：无法识别/不支持的数据就出现幻觉

## 二、Prompt Engineering

### 2.1 核心理解

> **Prompt = 与 AI 定好的正式沟通规矩**

| 原则 | 说明 |
|------|------|
| Prompt 是软约束 | 提高遵循概率，但不保证安全 |
| 指令分层 | 系统规则、用户目标、检索内容不能混 |
| 结构化输出 | 关键结果用 JSON/Schema，程序可验证 |
| 好的 prompt 让模型在合适时停下来 | 不是更自信，而是知道何时不确定 |

### 2.2 好 Prompt 的要素

| 要素 | 说明 |
|------|------|
| Role 角色 | 你是谁 |
| Task 任务 | 要做什么 |
| Context 背景 | 相关信息 |
| Input 输入 | 给定什么数据 |
| Constraints 约束 | 不能做什么 |
| Safety Rules 安全规则 | 什么操作要停止 |
| Output Format 输出格式 | 返回什么结构 |
| Examples 示例 | 长什么样的好回答 |
| Failure Handling 失败处理 | 不确定时怎么做 |

### 2.3 Prompt Injection 安全

> **你想到的案例：抖音数字人被注入 Prompt 后学鸭子叫**

| 攻击来源 | 用户输入、网页、文档、邮件、检索结果、外部数据库 |
|----------|---------------------------------------------------|
| **Agent 场景更危险** | Agent 能读私有信息、调用工具、执行高危操作 |

**防护措施：** 外部内容标记为不可信 → 工具调用参数校验 → 敏感动作 allowlist → 高风险 human approval → 不把密钥/大额资产权限给模型

### 2.4 AI×Web3 中 Prompt 的工作流

```
Prompt 定义任务和输出格式
  → Context 提供可信数据和来源边界
  → Model 生成解释或候选动作
  → Code 校验 schema 和业务规则
  → Guard/Simulation 检查链上影响
  → Human Check 确认高风险动作
```

## 三、Context & RAG

### 3.1 Context 上下文

> **定义：模型这一次能看到、能使用、并据此做判断的信息空间**

| 核心认知 | 说明 |
|----------|------|
| 模型不聪明可能不是模型的问题 | 是 Context 设计混乱 |
| Context 不是大杂烩 | 系统规则/用户目标/历史状态/工具结果/外部文档要分区 |

**Context 设计三原则：**

1. **可信来源显式标注** — 系统状态、用户输入、检索文档、工具结果分开
2. **上下文可刷新** — 状态、权限、价格不能长期缓存
3. **记忆可撤销** — 偏好不是隐藏权限，不能变成永久身份假设

> ⚠️ Context Window 越大 ≠ 越好。配合检索/摘要/结构化/来源标注/可信度分层才靠谱。

### 3.2 Agent Context 五层模型（AI×Web3）

| 层级 | 放什么 | 优先级 | 可替代 |
|------|--------|--------|--------|
| **指令层** | 系统规则、安全边界 | 最高 | ❌ |
| **任务层** | 用户本次目标、参数 | 高 | ❌ |
| **事实层** | 链上查询、余额、合约状态 | 最高可信 | ❌ |
| **知识层** | 文档、SDK、EIP、历史案例 | 参考 | ✅ |
| **记忆层** | 用户偏好、常用配置 | 辅助（不可替代授权）| ✅ 需重确 |

**分层的目的：** 权限控制 → Prompt Injection 防护 → 审计追溯

### 3.3 Memory 记忆

- 让 Agent 更顺手，不用每次都重新说明背景
- ⚠️ 涉及身份/权限/资产的记忆必须重新绑定当前会话和授权
- Memory 不能替代实时授权

### 3.4 RAG 检索增强生成

> **要让 AI 回答每一句话都做到有据可查**

```
用户提问 → 检索相关资料 → 筛选证据 → 把证据交给模型 → 模型基于证据回答
```

**RAG 解决两个问题：** 模型知识过期 + Context Window 有限

**RAG 三原则：**
1. 搜到的内容不一定真 → 判断来源/时间/适用性
2. 每个关键点能点回去看原文 → 引注
3. 搜不到靠谱资料直接说"不知道" → 不硬编

### 3.5 RAG 核心组件

| 组件 | 是什么 | 核心要点 |
|------|--------|----------|
| **Chunking 分块** | 长文档切成可检索片段 | 按自然结构切，保留标题/URL/版本/时间 |
| **Vector DB 向量库** | 存 embedding 的智能档案柜 | 必须配 metadata（来源/版本/链条/时间/可信度） |
| **Retriever 检索器** | 根据问题取候选材料 | 不能只看意思像，要综合过滤条件 |
| **Rerank 重排序** | 对候选重新排优先级 | 变慢变贵，高风险场景值得加 |
| **Citation 引用** | 把结论连回来源 | 不是装饰，是验证入口 |

### 3.6 RAG 在 AI×Web3 中

- **只提供信息，不替用户做决定**
- RAG 结果要影响链上动作时，必须接 simulation + policy + human check
- 文档说"这个函数存在"≠ 用户该签 — 能力 vs 权限是两个问题

## 四、Agent 智能体

### 4.1 核心理解

| 普通 LLM | Agent |
|----------|-------|
| 你问一句，答一句 | 给目标，自己拆步骤/查资料/调工具 |
| 只会"说" | 能"做" |
| 说错只是说错话 | 做错就真干了坏事 |

> **Agent 不是"想干啥就干啥"，是被上了很多锁的执行循环。**

**真 Agent 的六要素：**
1. 目标（要达成什么）
2. 工具（能调什么 API）
3. 状态（走到哪一步了）
4. 权限（读/写边界）
5. 停止条件（什么时候必须停）
6. 审计（每步都能查）

### 4.2 Tool Use 工具调用

- 工具让 Agent 从"会回答"变成"能做事"，也是最大风险入口
- ⚠️ 能读网页的 Agent 被恶意网页骗 → 能写入的 Agent 被恶意网页骗 = 直接删库

**工具设计的"三权分立"：** 模型提议 → 系统限制 → 用户批准

### 4.3 Planning / State / Reflection / Multi-Agent

| 概念 | 核心 | 风险 |
|------|------|------|
| **Planning** | 把目标拆成步骤 | 计划是建议路线，不是授权书 |
| **State** | Agent 的进度存档（存外部，不存模型记忆） | 模型记忆会忘/会压缩/会瞎编 |
| **Reflection** | Agent 自查中间结果 | 模型可能给自己的错误找理由 |
| **Multi-Agent** | 多个 Agent 分工 | 责任边界模糊、错误扩散、更难调试 |

### 4.4 AI×Web3 Agent 安全架构（8 步）

| 步骤 | 动作 | 谁负责 |
|------|------|--------|
| 1 | 用户提出目标和约束 | 用户 |
| 2 | Agent 读上下文 + 生成计划 | Agent |
| 3 | 系统拆"只读"和"写入" | 系统 |
| 4 | 只读自动执行 | 系统 |
| 5 | 写入 → Policy 检查 | 系统 |
| 6 | Simulation 展示链上影响 | 系统 |
| 7 | 用户确认高风险 | 用户 |
| 8 | 钱包执行 + 日志 | 链上 |

> **Web3 铁律：只读可自动，写入要模拟+审批+审计，绝不给 Agent 大额资产+模糊目标**

## 六、AI 框架（Framework）

> **核心认知：框架管理工程复杂度，不管模型对不对**

| 原则 | 大白话 |
|------|--------|
| 框架是系统边界的表达，不是智能本身 | 框架只是把设计好的流程"写出来"，不会替我想产品逻辑 |
| 先理解工作流，再决定用不用框架 | 别一上来就套框架，先画清楚：输入、状态、工具、输出、评估、失败路径 |
| 简单链路先保持简单 | 问一次→检索一次→回答一次，不需要复杂框架 |
| 长流程需要显式状态 | 多步任务必须有可查询的 state（状态存外面） |
| 框架要能退出 | 如果框架让我很难换模型/换向量库/换部署方式，长期成本很高 |

**💡 我的洞察：** 选错框架的后果不是"跑不起来"，而是"跑起来之后，改不动、测不了、换不掉"。

### 框架对比

| 框架 | 定位 | 什么时候用 | 注意 |
|------|------|-----------|------|
| **LangChain** | 乐高式组件库，快速拼能力 | 学习、快速原型 | 容易过早抽象，排查问题难 |
| **LangGraph** | 工作流+状态机，图结构 | 多步任务、分支/循环/重试/人工确认 | 简单问答太重 |
| **OpenAI Agents SDK** | 构建带工具/handoff/护栏的 Agent | 多 Agent 交接、工具调用 | 自己定义工具和失败条件 |
| **DSPy** | 程序化优化 prompt | 有数据集+指标的任务（分类/抽取/问答） | 不适合简单文案 |
| **Hermes** | 面向工具调用和结构化输出 | Tool calling、JSON 输出、长上下文 | 不看榜单分数，看稳定输出 |
| **Learning Agent** | 从反馈/日志中自我改进 | 长期运营的系统 | 数据污染风险，先进评估闭环再上线 |

### AI×Web3 分工

| 层级 | 负责什么 |
|------|---------|
| AI Framework | 管理 prompt、工具、状态、评估、追踪 |
| Web3 基础设施 | 管理账户、签名、合约、交易、链上状态 |
| 产品层 | 定义用户目标、权限边界、确认流程、失败处理 |

## 七、Vibe Coding 氛围编程

> **我理解了：Vibe Coding 是一种「分工明确的 AI 协作写代码方式」，不是凭感觉写代码**

**分工：**
- 我负责：方向、约束、验收（"做什么、不能做什么、做成什么样算好"）
- AI Agent 负责：生成代码、修改文件、搜索文件、跑测试、解释错误、提 PR

**核心不是让 AI 替我写所有代码，而是把工程反馈循环压短**——原来几分钟一个回合，现在几秒。

**速度变快后真正的问题：**
- 我能不能清楚描述任务（不模糊）？
- 我能不能审查 AI 生成的结果？
- 我能不能控制改动范围（别让它乱改文件）？
- 我能不能验证没有引入新 bug？

| 原则 | 大白话 |
|------|--------|
| 任务要小 | 越具体、越有边界，Agent 产出越容易审查 |
| 上下文要准 | 让 Agent 看对文件、看约束，比写长篇需求更重要 |
| 验证要硬 | 跑测试、查类型、看日志，比"看起来对"可靠 |
| 给 AI 划施工围栏 | 哪些文件能改、哪些命令能跑、什么时候必须停下来让人确认 |

**适合：** 搭原型 / 修小 bug / 补测试 / 写脚本 / 重构局部  
**不适合：** 无边界重写整个项目

### 关键工具

| 工具 | 价值 |
|------|------|
| **Claude Code / Codex CLI** | 把 AI Agent 放进本地开发环境，能读文件、编辑、运行命令 |
| **Model / API Config** | 不是只看最强模型，要管上下文窗口、权限、成本、日志 |
| **GitHub / gh** | Agent 能看 issue、生成 branch、读 PR diff、写提交信息——但版本控制仍是我审查的关键线 |
| **CLI / Script** | Agent 适合生成数据迁移、检查脚本、批量修改——注意 dry run |
| **Repo Workflow** | 别让 AI 绕过流程，更好的方式是让 AI 参与流程 |

## 八、MCP 模型上下文协议

> **MCP = AI 界的 USB-C 标准。不是让模型更聪明，是让模型"接工具"的方式统一**

**解决的问题：** 每个 AI 应用自己写一套接工具的代码，MCP 把这层变成标准协议。AI 客户端和工具 Server 只要都遵守 MCP，就能互相协作。

| 组件 | 说明 |
|------|------|
| **MCP Server** | 提供能力的那一侧，暴露工具/资源/提示词给 AI 客户端 |
| **MCP Client** | 连接模型和 Server 的那一侧，负责发现工具、转发调用、权限提示 |
| **Tool Schema** | 工具的"说明书"：叫什么、什么时候用、参数格式、只读还是写入 |
| **Permission** | 方便≠安全——至少区分只读/写入、会话授权/长期授权、是否需确认 |

**Server 设计重点：** 暴露范围要清楚（别暴露整个硬盘）、只读工具隔离写入工具、参数格式要明确、错误要返回不静默失败

**AI×Web3 中的定位：** MCP 可以标准化区块浏览器查询、合约文档检索、RPC 调用等——但不能替代钱包权限、交易模拟、签名确认。**MCP 管工具调用，Web3 账户系统管最终权限边界。**

## 九、Evaluation 评估

> **从"凭感觉觉得不错"变成"能持续可靠地判断系统好坏"**

**核心问题：** 我改了一句 prompt，A 类问题变好了但 B 类变差了——怎么知道这次改动到底有没有进步？

| 原则 | 大白话 |
|------|--------|
| 不能被重复测量的 AI 行为，就不能被稳定改进 | 先测任务，不只测模型 |
| 先保住关键失败场景 | 高风险错误先放回归测试集 |
| 评估要贴近产品 | 离真实用户越远，评估越像自欺欺人 |

### 五个核心概念

| 概念 | 大白话 | 核心要点 |
|------|--------|----------|
| **Harness** | 跑 eval 的测试架子 | 喂样本→调系统→打分→记录，价值是可重复 |
| **Golden Set** | 精挑细选的测试样本集 | 早期 30~100 条高质量样本覆盖常见/边界/高风险/历史 bug |
| **LLM-as-Judge** | 用模型当评委 | 适合开放答案（摘要质量、引用准确性）——不是万能，需要定期校准 |
| **Regression** | 防止旧问题复发 | 每修一个 bug 就变成回归样本，每次改动都重跑 |
| **Observability** | 线上真实可观测 | 记录输入来源、检索结果、工具调用、用户反馈、成本延迟 |

**AI×Web3 中评估更关键：** 交易解释是否准确、风险是否漏报、工具调用是否越界、能否拒答、能否识别 Prompt Injection

## 十、Fine-tuning 微调

> **微调不是"效果不好就训练一下"，而是先排查 prompt、上下文、输出格式、模型能力，最后再想微调**

| 原则 | 大白话 |
|------|--------|
| 微调改的是模型行为分布，不是产品系统边界 | 能让模型更习惯我的数据，但不会自动带来事实正确和权限安全 |
| 先有 eval，再谈微调 | 没有评估，不知道微调后是好还是只在样本上变顺了 |
| 先修数据，再修模型 | 坏数据会把坏习惯训练得更稳定 |
| 别用微调存实时知识 | 实时价格、文档更新放 RAG 或工具，别硬训练进模型 |

### 核心概念

| 概念 | 做什么 |
|------|--------|
| **SFT** | 给模型 (输入, 期望输出) 例子让它模仿，适合统一格式/风格/术语 |
| **LoRA** | 省钱省显存的微调，训一小部分参数插到原模型 |
| **PEFT** | 参数高效微调总称，少量参数适配新任务 |
| **Dataset** | 核心资产不是问答对多，是质量可控——训练/验证/测试/回归集必须分开 |
| **Overfitting** | 把训练数据背太死，换新问题就崩——样本太少、风格太单一、训练轮数太多都会导致 |

**AI×Web3 中：** 微调能让模型更懂我的格式和习惯，但不能替代链上实时查询、合约审计、交易模拟、钱包权限控制、用户确认。

## 十一、Inference 推理服务

> **推理不是点一下按钮，而是在延迟、成本、隐私、稳定性之间做权衡**

**产品一旦有真实流量，问题就来了：** 响应太慢、成本爆炸、模型升级输出格式变了、隐私数据能不能送第三方 API、本地模型够不够用、服务崩了怎么降级？

| 原则 | 大白话 |
|------|--------|
| 推理核心 = 在约束条件下交付可用答案 | 不只是跑出答案，要快、不贵、稳定、安全 |
| 同一个模型不同部署方式表现完全不同 | 选的不只是模型，是一套权衡 |
| 质量有代价 | 更强模型 = 更高成本/更长延迟/更复杂部署 |
| 服务要可替换 | 把模型调用封装好，能 fallback、灰度、评估 |

### 核心概念

| 概念 | 做什么 | 注意 |
|------|--------|------|
| **API Model** | OpenAI/Anthropic 云端调用 | 还要处理速率限制、超时、重试、日志脱敏、账单 |
| **Local Model** | 自己电脑/私有环境跑 | 隐私好+成本可控，但受限于显存/并发/量化 |
| **Quantization** | 降精度让模型更小更快 | 体积↓速度↑，但质量？必须实测自己任务 |
| **Serving** | 模型服务化 | 并发/队列/流式/缓存/监控/失败处理/版本灰度 |

**AI×Web3 中特别要求：** 链上动作不可逆，推理必须留审计记录——用了哪个模型版本、输入来源、输出内容、是否触发工具调用、失败怎么处理。

## 十三、关键感悟

1. **AI 是脑洞工具，不是决策工具** — 模型能生成很棒的思路（"脑洞很不错"），但最终判断需要人
2. **结构化输出不是装饰，是安全机制** — 只有结构化输出才能被代码校验、拒绝、追溯
3. **Agent 的安全 = 三权分立** — 模型提议、系统限制、用户批准，三者缺一不可
4. **Context 设计比模型聪明更重要** — 不是模型笨，是上下文设计混乱
5. **Web3 铁律不可逾越** — 只读自动、写入审批、大额人工确认，这是底线
