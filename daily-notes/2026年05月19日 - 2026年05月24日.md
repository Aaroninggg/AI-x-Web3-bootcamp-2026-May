	做风险拦截、交易模拟、权限管控、重要操作人工审核、全程留痕记录，防止误操作和被骗。

# Prompt
Prompt不只是随口提问，是与 AI 定好的正式沟通**规矩**，作用为将一个模糊任务拆成模型可以稳定执行的工作说明。
	任务目标、输入边界、输出格式、不确定处理、安全规则

**好的 prompt 不是让模型更自信，而是让模型在合适的时候停下来。**

prompt中的指令层要讲清楚规则、目标；输出层要让机器可验证；遇到高风险问题要停下来。

### instruction 任务规则，
	角色Role、 要完成的任务、不能做的事情、不确定事情的处理方式、最终输出是什么样的。

### Few-Shot 少量举例，示意模型应该怎么做。让模型看着例子学规则。

### Structured Output 结构化输出
	就是返回值是可验证的，符合我要求结构的结果。
	我更倾向于JSON、Schema
	结构化输出不是为了好看，而是为了让后续代码能检查、拒绝、记录和回归测试。

### Prompt injection 提示词注入攻击
通过输入内容诱导模型忽略原本规则，执行危险行为，
这让我想到了抖音数字主播在直播的时候，被注入了Prompt，数字人开始学鸭子叫

攻击来源可能是：用户输入、网页、文档、邮件、检索结果、外部数据库等

如果Agent被注入，Agent可以读取私有信息，调用工具，等高危行为

一句 不要被攻击 无法抵挡Prompt injection，更可靠方法：
	- 把外部内容标记为不可信数据；
	- 工具调用前做参数校验；
	- 敏感动作必须经过 allowlist；
	- 高风险操作必须 human approval；
	- 不把密钥、主权限、不可撤销动作交给模型。

# Context 上下文
定义：是模型这一次能看到、能使用、并据此做出判断的**信息空间**
上下文包含：当前会话内容，系统规则与开发规则、长期记忆（用户偏好、项目背景等）
、调用的工具。
Context 不是简单地把很多资料塞给模型，而是要把不同类型的信息分清楚：
	- 哪些是系统规则；
	- 哪些是用户目标；
	- 哪些是历史状态；
	- 哪些是工具返回结果；
	- 哪些是外部文档或网页内容；
	- 哪些信息可信，哪些信息只是参考。

LLM 本身不会自动知道你的数据库、文件、API、项目文档、用户历史或链上状态。
它只能根据当前上下文回答问题。
	所以，如果上下文：
		- 缺失；
		- 过期；
		- 来源不清；
		- 可信度不明；
		- 权限边界混乱；
	模型就很容易给出错误答案。
很多是够不是模型不聪明，是Context设计混乱。
### Context 设计关键原则
Context 设计要注意三点：
1. **可信来源要显式标注**  
    系统状态、用户输入、检索文档、工具结果要分区放置，不能混成一段文本。
2. **上下文要可刷新**  
    状态、配置、权限、价格、版本、任务进度等信息可能会变，不能长期缓存后继续使用。
3. **记忆要可撤销**  
    用户偏好和历史任务可以帮助模型更懂用户，但不能变成隐藏权限，也不能变成永久身份假设。
Prompt 告诉模型怎么做，Context 决定模型依据什么做；

### Context Window：模型一次能看的最大范围
模型一次请求中能处理的最大上下文长度，窗口越大，能放进去的内容越多
长上下文常见问题是：
- 模型看见了很多内容，但没有抓住重点；
- 无关信息太多，干扰判断；
- 模型引用了旧文档或过期段落；
- 关键信息被淹没在大量文本中。

所以 Context Window 不是越大越好。实际产品里，要配合：
- 检索；
- 摘要；
- 结构化数据；
- 来源标注；
- 可信度分层。
Context Window 决定模型最多能看多少，但 Context 设计决定模型能不能看对、用对

### Context Engineering：上下文工程
它不是简单地把资料全部塞给模型，而是要决定：
- 哪些数据源可以用；
- 哪些信息应该排在前面；
- 哪些内容需要裁剪；
- 哪些内容必须标注来源；
- 哪些外部内容是不可信的；
- 哪些状态必须每次重新查询；
- 哪些信息可以作为历史参考。

一个稳定的工具型 Agent，上下文通常应该包括：
- 当前任务状态；
- 用户原始意图；
- 工具返回结果；
- 相关日志和证据；
- 可信数据来源；
- 外部检查结果；
- 系统禁止事项；
- 输出 schema。

Context Engineering 的核心是信息治理：让模型知道什么是规则、什么是事实、什么是参考、什么是不可信内容。

### Memory：跨请求保留的信息、记忆
Memory 是模型或系统在不同请求之间保留下来的信息。
	- 用户偏好；- 历史任务；- 常用钱包；- 项目配置；- 上次分析结果；- 常用输出格式。
	- 优点：是让 Agent 更顺手，不需要用户每次都重新说明背景。

Memory 不能替代实时授权。**所有涉及身份、权限、资产或外部副作用的记忆，都必须重新绑定当前会话和当前授权。**

Memory 可以提升体验，但不能变成隐藏权限，更不能替代用户的实时确认。


## Knowledge Base：外部知识库
Knowledge Base 是系统可以检索的外部知识集合。作用是解决模型知识过期的问题。
Knowledge Base 不是天然可靠的，需要定期维护。否则 RAG 只会把旧的、错的、不适用的信息更稳定地喂给模型。
**Knowledge Base 可以补充模型知识，但必须管理版本、来源和有效性，否则知识库本身也会变成错误来源。**

### Sumup 涉及较多的Web3知识，需要消化
Context 在 AI × Web3 里，是模型连接真实链上世界的入口。
AI 本身负责推理、解释和生成判断；Web3 系统负责提供真实状态、资产变化和最终结算。

### 一个靠谱的 Agent Context 应该分层

在 AI × Web3 场景里，Context 最好不要混在一起，而是按层级管理。

#### 1. 指令层

这一层放最高优先级规则，例如：

- 系统规则；
- 工具使用规则；
- 禁止事项；
- 安全边界；
- 输出格式要求。

它决定模型**必须遵守什么**。

#### 2. 任务层

这一层放本次用户的具体目标，例如：

- 用户想分析哪笔交易；
- 用户想查询哪个钱包；
- 当前网络是 Ethereum、Base、Arbitrum 还是 Solana；
- 本次任务的参数和范围。

它决定模型**这次要做什么**。

#### 3. 事实层

这一层放最关键的真实状态，例如：

- 链上查询结果；
- 钱包余额；
- 合约状态；
- 工具返回结果；
- transaction simulation；
- gas、slippage、approval 变化；
- 实际资产流入流出。

它决定模型**依据什么事实判断**。

#### 4. 知识层

这一层放背景知识，例如：

- 协议文档；
- SDK 文档；
- EIP / 标准；
- 官方说明；
- 论坛讨论；
- 历史案例；
- 风险知识库。

它帮助模型理解背景，但可信度通常低于实时工具结果。

#### 5. 记忆层

这一层放用户或项目的长期信息，例如：

- 用户偏好；
- 常用项目配置；
- 常用网络；
- 常见输出格式；
- 历史分析习惯。

但记忆层不能替代当前授权。尤其是钱包、资产、交易、权限相关内容，必须重新确认。

### 为什么要分层
Context 分层之后，系统才更容易做三件事：

1. **权限控制**  
    模型知道哪些信息只能参考，哪些信息可以用于判断，哪些动作必须人工确认。
2. **Prompt Injection 防护**  
    外部文档、网页、论坛内容不会被误当成系统指令。
3. **审计与追溯**  
    当模型给出一个判断时，可以追踪它依据的是链上事实、工具结果、文档，还是历史记忆。


## [最小实践](https://aiweb3.school/zh/handbook/ai/context/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

为“钱包授权检查 Agent”设计一份 context spec。

选择一个场景：用户问“这个 dApp 要我 approve，可以签吗？”你需要列出模型回答前必须拿到的上下文：

- chain id 和当前区块
- token 合约、spender 地址、approve 数量
- 用户当前 allowance 和余额
- spender 是否在可信列表
- simulation 或静态检查结果
- dApp 页面提供的说明，标记为不可信外部内容
- 用户本次意图

再写清楚哪些字段必须实时查询，哪些可以来自缓存，哪些不能被模型当成事实。

# RAG  检索增强生成 Retrieval-Augmented Generation
RAG 是让模型基于外部证据回答问题的一套机制，不只是接一个向量数据库
RAG是一个知识库，有RAG，那么AI回复的每一句话都要做到有据可查。

先检索相关资料 → 再筛选证据 → 再把证据交给模型 → 最后让模型基于证据回答

RAG解决2个问题，模型知识会过期、Context Windows有限制
在用户提问时，从知识库里取回相关材料，让模型基于这些材料回答，而不是只靠模型记忆。

而真正的RAG要求三点：
1. **搜到的内容不一定是真的**（可能是过期文档、错误版本、不适用于当前场景），所以系统必须先判断来源是否可靠、时间对不对、适用范围是否匹配。
2. **回答里的每个关键点，必须能点回去看原文**（比如“这句话出自你文档的第3页第2段”），而不是AI自己组织语言后你就没法验证了。
3. **如果搜不到靠谱的资料，AI必须直接说“我不知道”**，而不是硬编一个看起来合理的答案。
###  Chucking 文档切块

长文档切成可检索片段，可以理解为是把一本很厚的书拆成一张一张的小卡片，方便以后找东西。
	**切得太碎**：比如把一句话切成两半，意思就断了，
	**切得太大**：一张卡片上写了10个不同的问题，你问其中一个，结果AI把所有内容都塞给你，里面很多废话，浪费钱（token成本）。
	Chuck 就是一个切片
如果是将类似于技术文档、风险警告等信息加入RAG，切片不当会造成非常严重的后果。例如：可能切到“这个函数可以删文件”，但删文件的警告“仅在测试环境可用”被切到下一张卡片了。这样AI只拿到前半句，就很危险。

稳妥做法：按自然结构切，比如一个API说明就是一个完整块，**保留标题、URL、更新时间、版本号。**

### Vector DB 向量数据库
Ventor是用来储存 embedding的。
它可以理解是一个超级智能的档案柜，你问任何问题，都可以通过embedding的相关性找到答案。 但是：**意思相近 ≠ 答案正确**。
所以不能只存储embedding，还需要给每一个卡片打上Metadata：
Metadata：
	- **来源**：哪来的 
	- **版本**：哪个版本
	- **chain**：从哪条链路来的（谁导入的/哪个上游系统）
	- **更新时间**：什么时候更新的
	- **可信等级**：多可信
	- **是否废弃**：还能不能用

### Retriever 检索器
Retriever 是根据用户问题取回候选材料的组件。是“去仓库里找可能相关的材料”的那个组件
它可以很简单（比如只做向量搜索，意思相近就行），也可以很聪明（比如先按版本、时间、来源类型过滤，再搜索）。

**核心观点：不能只看意思像不像。**

要根据用户的问题，用户提出了很多过滤条件，就要把全部的条件都筛一遍。**而不是只搜意思相近的。**

### Rerank
Rerank 是在初步检索后，对候选材料重新排序，把更相关、更可信、更完整的内容排到前面。
是“初步搜出一堆候选材料后，再重新排个序，把真正有用的往前放”。

初步检索可能搜出 100 段内容，里面有高度相关的、沾边的、和完全不相关的。相似度分数可能很接近（比如 0.82、0.79、0.77），靠分数分不出好坏。Rerank 会用更精细的方法（比如更贵的模型、规则等）重新打分，把真正靠谱的排到第1-3位，后面那些就不怎么用了。

**代价：**  
Rerank 会变慢、变贵。如果只是个简单的 FAQ 问答（几十个常见问题），没必要加。但如果是“开发者文档问答”或“资产风险相关的问答”（答错了会出事故或亏钱），那值得加。

### Citation 引用
Citation 是把答案里的关键结论连接回来源。它不是装饰，而是用户验证答案的入口。**“这句话是凭什么说出来的？点一下就能看到原文”**。**它不是装饰品**，是让你能验证答案真伪的入口。

**好的引用至少告诉你：**
- 这句话出自哪份文档（能点回去看原文）
- 来源是不是官方的（还是某篇个人博客）
- 文档是什么版本、什么时候更新的
- 哪些结论是模型自己归纳总结的（原文没直接说）
- 哪些观点缺乏足够证据（模型在猜）

**反例（没有引用的RAG）：**  
AI 说“这个 API 已废弃，建议用 v3 的新接口”。你问“谁说的？文档在哪？”——AI 答不上来。你可能被坑了。

**核心金句（原文）：**

> 如果一个 RAG 系统不能展示来源，用户就很难判断答案是基于文档、基于旧知识，还是模型自己补出来的。

###  **RAG 在“AI + Web3”**的应用
#### RAG 的位置：知识库和模型之间

**人话解释：**  
RAG 不是知识库（存文档的地方），也不是模型（负责生成答案的那个 AI）。它夹在中间当“情报员”：

- 用户问问题 → RAG 去知识库里找相关材料 → 把材料和问题一起给模型 → 模型基于材料回答
它**不负责执行**（比如不负责签名交易、不负责转账、不负责投票），只负责**查资料和补上下文**。
#### 2. 常见应用（都偏“读”，不偏“写”）

|应用场景|大白话|
|---|---|
|协议文档问答|用户问“Uniswap V3 的费率有几个档位”，RAG 去白皮书/官方文档里找答案|
|合约接口解释|用户给一个函数签名 `swapExactTokensForETH`，RAG 查文档解释它干什么、参数什么意思|
|治理提案和论坛摘要|DAO 里有几百页的提案讨论帖，RAG 帮你总结成几句话：“这提案是要把 treasury 的 10% 换成 USDC”|
|审计报告检索|问“这个合约有没有重入漏洞”，RAG 去找审计报告里相关段落|
|SDK / API Copilot|写代码时问“这个方法的第二个参数是什么意思”，RAG 去查 SDK 文档|
|交易解释时补充项目背景|用户看到一笔未知交易“调用某合约”，RAG 查项目资料说“这项目是做跨链桥的，这交易可能是正常的跨链操作”|

所有这些场景的共同点是：**只提供信息，不替用户做决定**。
#### 3. 关键警告：RAG 结果要影响链上动作时，不能直接信

**人话解释：**  
RAG 找到的文档里说“这个函数可以调用”，**不等于**用户应该点“签名”按钮。

文档说的是“能力”：“这个按钮存在，按了会怎样”。  
但用户需要判断的是“权限”：“当前我该不该按？”

举例：

- 文档说 `withdrawAll()` 函数可以提取所有资金
- 但这个函数可能只能在特定时间、特定角色、特定条件下调用
- 或者用户连接的钱包根本不是管理员，调了也没用
- 或者这是个钓鱼合约，文档本身是假的


**所以原文说：** 当 RAG 结果要影响链上动作时，还需要接：

- **simulation**（模拟执行一下，看看结果是什么）
- **policy**（规则检查：用户有没有权限？当前环境允许吗？）
- **human check**（让人确认一遍，尤其涉及资产时）

## [最小实践](https://aiweb3.school/zh/handbook/ai/rag/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

做一个“协议文档 RAG 问答”的最小版本。

选择一个官方文档站，抓取 10 到 20 篇页面，按标题结构切 chunk，存入向量库。用户提问时，系统返回答案和引用。

至少测试三类问题：

- 文档中明确存在的 API 用法
- 文档中不存在的问题，要求系统拒答
- 旧版本或不确定版本的问题，要求系统提示需要核对版本

输出时强制区分：

- `answer`
- `sources`
- `uncertainties`
- `needs_version_check`

这个练习的重点是证据链，不是 UI。

# Agent 智能体
Agent 是能围绕目标持续调用工具、读取状态、调整步骤的 **AI 系统**。它的关键不在“像人”，而在于它能不能在明确权限和可审计流程里，把建议推进到行动。
	- **普通 LLM**：你问一句，它答一句。只会“说”，不会“做”。
	- **Agent**：你给一个目标（比如“帮我分析这个 DAO 提案安不安全”），它能自己拆步骤、查资料、调工具、写代码、生成草稿，甚至问你“下一步可以吗？”然后继续。

风险在于，普通模型答错，也就是说错话。Agent 做错就是真干了坏事：

**所以学 Agent 的重点不是“怎么让它更像人”，而是：**
	- 模型负责“提出想干什么”
	- 系统负责“限制能干什么”
	- 用户负责“批准高风险的事”

Agent 不是“想干啥就干啥”，而是 **被上了很多锁的执行循环**。必须有这六样东西才算真 Agent：
1. **目标**（要达成什么）
2. **工具**（能调什么 API）
3. **状态**（现在走到哪一步了）
4. **权限**（能读不能写？能写需要谁同意？）
5. **停止条件**（什么时候必须停下来）
6. **审计**（每一步都能查）

> 一个只有模型+工具、没有权限和停止条件的 Agent，本质上是“会说话的脚本执行器”，不是真 Agent。但是，
> 
> 这种**工具比回答更危险**：读数据、写数据库、发请求、改配置，风险完全不一样。读可以不问人，写必须严控。
> 
> **状态必须存外面**：不能只放在模型的记忆里（因为它会忘或瞎编）。要能回答“它为什么调了那个工具？用户同意了吗？”
> 
> **停止条件要明确**：目标达到了、钱花超了、信息不够、风险越界、用户拒绝——都要能停。

### Tool Use 工具调用

Tool Use 是 Agent 调用外部能力：搜索、数据库、API、代码执行、邮件、支付接口、内部系统等。
一句话解释就是：工具让 Agent 从“会回答”变成“能做事”。但工具也是最大的风险入口。
	**危险例子：**
	- 能读网页的 Agent 被恶意网页骗了 → 可能输出错误信息
	- 能写入数据库的 Agent 被恶意网页骗了 → 直接删库

**工具设计必须明确：**
- 输入格式是什么（输入 schema）
- 权限范围多大（权限范围）
- 是只读还是能写
- 是否会产生外部副作用
- 调用前要不要人确认
- 每次调用都要记日志（调用前后如何记录）


### Planning （规划）
Planning 是把目标拆成步骤。模型生成的计划只是“建议路线图”，不是“授权书”。越靠近高风险动作（如转账、改合约），计划越需要被系统规则拆开检查。

**好的计划要暴露：**
- 每步用什么工具
- 每步是读还是写
- 哪些可自动执行
- 哪些要用户确认
- 失败了能不能重试
- 怎么验证完成了

### State（状态）

State 是 Agent 当前任务的状态，包括用户目标、已完成步骤、工具返回、错误、预算、确认记录和最终输出。
相当于，Agent 当前任务的“进度存档”。包括：用户目标、已完成步骤、工具返回了什么、有没有报错、花了多少预算、用户确认了哪些、最终输出了什么。

**很多 Demo 的问题：** 只把状态存在模型的对话历史里（因为模型会忘、会压缩、会瞎编）。生产系统需要存在外部数据库里，能查、能恢复、能审计。

**有外部执行时（比如链上操作），状态还要记录：** 环境、版本、参数、工具调用结果、确认请求、撤销事件。

### Reflection（反思）
  
让 Agent 自己检查中间结果，发现问题就修正。比如“哦，信息不够，我再查一次”或“这个计划好像不合理，我改一下”。

**但注意：** 反思还是模型自己做的，模型可能会给自己的错误找理由（“我觉得这样没问题”）。

> 反思能提高质量，但不能替代外部验证。尤其是写入、授权、支付这类动作，不能只靠 Agent“自己觉得没问题”。

**总结：** 自我检查 = 辅助诊断；确定性检查 = 承载安全。

### Multi-Agent（多智能体）

多个 Agent 分工干活。比如：研究 Agent 读资料 → 开发 Agent 写代码 → 安全 Agent 审风险 → 执行 Agent 调工具。

**好处：** 适合复杂工作流。  
**坏处：** 协调问题放大——上下文传递丢了、责任边界模糊了、一个 Agent 的错误被另一个当成事实、工具权限扩散了。

> 做 Multi-Agent 前先问自己：多个 Agent 真的比一个 Agent 更简单吗？如果本来流程就不清楚，拆成多个角色只会更难调试。

### 在 AI x Web3 中的位置
Agent 夹在“模型能力”和“链上执行”之间。它能把用户目标拆成多步，但不能绕过账户、权限和结算规则。

**一个相对稳的 AI x Web3 Agent 架构（8步）：**

|步骤|干什么|谁负责|
|---|---|---|
|1|用户提出目标和约束|用户|
|2|Agent 读上下文 + 生成计划|Agent|
|3|系统把计划拆成“只读步骤”和“候选写入步骤”|系统|
|4|只读工具自动执行|系统|
|5|写入工具进入 policy 检查（权限、规则）|系统|
|6|Simulation 展示链上影响（模拟执行）|系统|
|7|用户确认高风险动作|用户|
|8|钱包/智能账户执行 + 日志记录|链上|

**最危险的设计（千万别做）：**

> 让 Agent 同时拥有：模糊的目标 + 广泛的工具 + 长期记忆 + 大额资产权限

这等于给一个不太靠谱的 AI 发了张无限额的黑卡。
Web3 铁律：只读可自动，写入要模拟+审批+审计，绝不给 Agent 大额资产+模糊目标

## [最小实践](https://aiweb3.school/zh/handbook/ai/agent/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

做一个“DAO 提案研究 Agent”的最小版本。

它只能执行只读动作：

- 读取提案正文
- 检索论坛讨论
- 总结支持和反对理由
- 标记缺失信息
- 输出投票前检查清单

不要让它直接投票。相反，要求它在输出里明确：

- 用了哪些来源
- 哪些结论证据不足
- 是否发现治理或资金风险
- 如果用户要投票，还需要人工检查什么

完成后，再设计一个权限升级版本：只有当用户明确授权，并经过投票交易 simulation 后，系统才允许生成投票交易草稿。

  

# AI框架 Framework

AI 框架是把模型、工具、状态、检索、评估和部署组织成一个可维护的系统。

选错框架的后果不是“跑不起来”，而是“跑起来之后，改不动、测不了、换不掉”。

  

刚开始做 AI 应用，直接调 API 就够了

进入产品阶段：你会遇到一堆工程问题——prompt 改了好几版怎么回滚？工具调用的输入输出格式怎么统一？Agent 执行到一半崩了怎么恢复？用户反馈怎么变成测试用例？线上行为怎么追踪？

  

框架解决的是**工程组织问题**，不解决模型对不对的问题。学框架时最重要的不是追流行，而是搞清楚：**这个框架到底帮你管理了哪一层复杂度？它又把哪些复杂度给“藏起来”了（藏起来的东西可能以后会坑你）？**

  

框架是系统边界的表达，不是智能本身 | 框架只是把你设计好的流程“写出来”，它不会替你思考产品逻辑。

先理解工作流，再决定用不用框架 | 别一上来就套个框架，然后让产品逻辑去迁就框架。先画清楚：输入、状态、工具、输出、评估、失败路径。

简单链路先保持简单 | 如果只是“问一次 → 检索一次 → 回答一次”，不需要复杂框架。

长流程需要显式状态 | 多步任务、工具调用、人工确认、失败恢复，必须有可查询的 state（又回到“状态存外面”那个概念）。

框架要能退出 | 如果某个框架让你很难换模型、换向量库、换部署方式，长期成本会很高。

  

### LangChain

  

**最常见的框架**，覆盖模型接入、prompt、工具、检索、Agent、输出解析等。

  

像一套“乐高组件库”，你可以快速把各种能力拼起来。适合学习，也适合快速原型。

  

**注意：** 容易“过早抽象”——还没搞清楚自己的工作流，就套了一堆 Chain 和 Agent，后面排查问题会很难。

**定位：** 帮你组合能力，不替你决定产品边界。

  

### LangGraph

  

**偏向工作流和状态机**，把任务表示成图（graph）：节点执行动作，边控制流程，state 记录过程。

  

如果你的任务像“流程图”一样有分支、循环、重试、人工确认，用 LangGraph 比用一堆 prompt 历史更可靠。

  

**什么时候用？**

- 简单问答 → 太重

- 多轮工具调用、重试、人工确认、失败后从中断处恢复 → 适合

  

> 实用判断：一旦你开始关心“任务走到哪一步了、能否恢复、失败后从哪继续”，就该考虑 graph / state machine。

  

### OpenAI Agents SDK

  

**用来构建带工具、handoff（任务交接）、guardrails（护栏）、追踪的 Agent 应用。**

OpenAI 官方出的 Agent 框架，帮你把 Agent 的常见工程问题（多 Agent 之间怎么交接、工具调用怎么结构化、运行时怎么追踪）做成可组合的结构。

  

**关键：** 框架帮你执行流程，但你要自己定义：哪些工具可用、哪些动作需要确认、什么输出算失败。

  

### DSPy

  

**把 prompt / LM 流程写成可优化的程序**。你不手调 prompt，而是定义输入输出、模块和评估指标，然后让优化器自动改进 prompt 或调用策略。

  

传统做法：手调 prompt 直到效果满意。

DSPy 做法：你给我一批数据（输入+正确答案）+ 评估指标（比如准确率），我自己去试各种 prompt 写法，找到最好的。

  

**什么时候用？**

  

- 简单文案写作 → 没必要

- 有明确数据集、评估指标、可重复任务（分类、抽取、问答、推理链）→ 有价值

  

> 核心启发：不要只凭感觉调 prompt，让任务、数据和指标进入系统。

### Hermes

  

**这里不是指通用框架，而是指“面向工具调用和结构化输出的模型/Agent 生态”。**

  

框架不是唯一的抽象层。**模型本身的能力**也很重要：模型是否擅长 tool calling、JSON 输出、长上下文、推理。能力强的模型能减少你在框架里写的大量解析和兜底代码。

  

**看这类模型时，不要只看榜单分数**，要看它能否稳定地产生你需要的结构化输出和工具调用格式。

  

### Learning Agent（具备学习能力的 Agent）

  

**系统可以从反馈、日志、评估结果或用户修正中改进。** 这里的“学习”不一定是重新训练模型，也可以是更新 prompt、调整检索器、补充规则、改进测试集。

  

让系统越用越好。但很容易踩坑：**直接把线上用户反馈当成真理去改行为**——会导致数据污染、越权学习、不可解释的变化。

  

**正确流程：**

  

1. 记录失败样本

2. 人工（或规则）标注失败原因

3. 加入回归测试集

4. 修改 prompt、检索、工具或模型配置

5. 通过测试后再上线

  

> 学习能力要先进评估闭环，再进入生产系统。

  

## 在 AI x Web3 中的位置

框架负责把模型能力接到产品流程里：读上下文、调工具、生成动作、记录追踪、进入评估。

  

**但是框架不能替代 Web3 侧的权限、签名、交易模拟和账户规则。**

  

**清晰的分工：**

  

| 层级 | 负责什么 |

| ---------------- | --------------------- |

| **AI Framework** | 管理 prompt、工具、状态、评估、追踪 |

| **Web3 基础设施** | 管理账户、签名、合约、交易、链上状态 |

| **产品层** | 定义用户目标、权限边界、确认流程、失败处理 |

  

> 框架可以组织 Agent，不能替用户承担资产风险。

  

## [最小实践](https://aiweb3.school/zh/handbook/ai/frameworks/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

  

做一个“文档问答 + 工具调用”的最小框架对比。

  

同一个任务，用两种方式实现：

  

- 直接调用模型 API：用户问题 + 检索结果 + JSON 输出。

- 使用一个框架：同样的输入输出，但加入 tool schema、trace、失败重试和测试样本。

  

对比四件事：

  

- 哪个版本更容易读懂？

- 哪个版本更容易加工具？

- 哪个版本更容易定位错误？

- 哪个版本更容易写回归测试？

  

这个练习的重点不是选冠军，而是看清框架到底帮了你什么。

  
  
  

# 氛围编程（Vibe Coding）

  

**Vibe Coding（氛围编程）**一种人和 AI Coding Agent 配合写代码的新工作方式。

Vibe Coding 不是“凭感觉写代码”。恰恰相反，它是一种**分工明确的协作方式**：

  

- **人负责**：方向、约束、验收（“我们要做什么、不能做什么、做成什么样算好”）

- **AI Agent 负责**：生成代码、修改代码、搜索文件、跑测试、解释错误、甚至提 PR（Pull Request）

  

核心不是让 AI 替你写所有代码，而是**把工程反馈循环压短**——过去你要自己搜文件、改代码、跑测试、看报错、再改……现在 Agent 能并行做很多事，你只需要快速审阅和决策。

  

AI Coding Agent 已经能读代码、改文件、跑测试、解释错误、生成 PR。这会明显改变开发速度。但**速度变快后，真正的问题不再是“代码写得慢”，而是：**

  

- 你能不能**清楚描述任务**（不模糊）？

- 能不能**审查 AI 生成的结果**（它改的对不对）？

- 能不能**控制改动范围**（别让它乱改不相干的文件）？

- 能不能**验证没有引入新 bug**（靠测试和检查）？

  

> AI 能帮你写更多代码，但不能替你承担工程判断。

  

| 原则 | 大白话 |

| --------- | ------------------------------------------------------------ |

| 把工程反馈循环压短 | 原来改一行代码要几分钟，现在几秒钟就能看到 AI 改完+跑测试的结果。但如果没有测试、审查、版本控制，速度只会放大混乱。 |

| 任务要小 | 越具体、越有边界，Agent 越容易产出你可审查的结果。别让它“重写整个项目”。 |

| 上下文要准 | 让 Agent 看对文件、设计约束、错误输出，比写长篇大论的需求文档更重要。 |

| 验证要硬 | 跑测试、类型检查、构建、看截图或日志，比“看起来对”可靠得多。 |

  

### Vibe Coding 本身

  

一种快速和 AI 共同探索代码的方式。你描述目标，Agent 生成方案或 patch（补丁），你通过运行、检查、追问和迭代来收敛。

  

**适合：**

  

- 搭原型

- 修小 bug

- 补测试

- 写脚本

- 重构局部模块

- 解释陌生代码

  

**不适合：** 无边界地“重写整个项目”。范围太大，Agent 容易改到不该改的地方，或者引入看似合理但不符合项目约束的抽象。

  

---

  

### Claude Code / Codex CLI

  

这两类工具把 AI Agent 放进你的本地开发环境，让它能读文件、编辑文件、运行命令、理解测试输出。

  

**关键价值：** 不是“聊天”，而是**能在你的 repo 里真正行动**。

  

**正因为能行动，你需要明确：**

  

- 哪些文件可以改

- 哪些命令可以跑

- 是否允许安装依赖

- 是否允许访问网络

- 什么时候必须停下来让人确认

  

> 本质：给 AI 划一个“施工围栏”。

  

---

  

### Model / API Config（模型和 API 配置）

AI Coding 不是只看“哪个模型最强”。你还要管理一堆工程细节：模型选哪个、API key 怎么存、上下文窗口多大、工具权限怎么设、成本怎么控制、日志怎么记录。

  

**常见问题：**

  

- 本地和 CI（持续集成）用的模型不一样 → 结果不一致

- API key 泄漏到日志或代码提交里 → 安全风险

- 上下文太长导致成本失控（比如一次传整个项目）

- Agent 用了不该用的工具权限

- 输出风格和项目规范不一致

  

> 好的配置应该能被团队复用，而不是每个人在自己电脑上随手调。

  

**提到 OpenCLI：** 它把网站、浏览器会话、Electron 应用、本地二进制工具统一成可被命令行调用的接口，让 AI Agent 可以通过稳定命令去使用工具，而不是每次都临时操作页面。

  

---

  

### GitHub / gh

  

AI 生成的代码不能只停留在聊天框里。你要把它放回版本控制、issue、PR 和 review 流程中。

  

**Agent 可以帮你：**

  

- 看 issue

- 生成 branch

- 读 PR diff

- 写提交信息

- 整理 review 说明

  

**但版本控制仍然是人类审查的关键线。**

  

**实用原则：** 让 Agent 多做**局部 patch**（小补丁），少做**不可追踪的大改动**。每次改动后至少看：

  

- `git diff`（改了什么）

- 修改文件列表

- 测试结果

- 有没有不小心提交的密钥、日志、构建产物

  

---

  

### CLI / Script（命令行/脚本）

  

很多开发任务不需要完整应用界面，写 CLI 或脚本更快。Agent 很适合帮你生成一次性脚本、数据迁移、检查脚本、批量修改。

  

**但脚本有两个边界：**

  

1. 读写文件前要确认范围（别删了不该删的）

2. 会修改外部状态的脚本要先 **dry run**（模拟运行，看会做什么）

  

> 如果脚本会删文件、发请求、改数据库、提交交易、调生产 API，就不能只靠 Agent 判断。

  

---

  

### Repo Workflow（仓库工作流）

把 AI Coding 接进正常的工程流程：issue → branch → commit → test → review → merge → release。

  

**不要让 AI 绕过这些流程。** 更好的方式是让 AI **参与**这些流程：

  

- 从 issue 提炼任务边界

- 搜索相关文件

- 生成最小 patch

- 跑测试并解释失败

- 补充 changelog 或文档

- 写 PR summary 和验证说明

  

> AI Coding 的质量，最终还是要落到 repo workflow 里。

  

## [最小实践](https://aiweb3.school/zh/handbook/ai/vibe-coding/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

  

选择一个小功能，让 AI Coding Agent 完成一轮完整工程闭环：

  

1. 写清楚任务边界和不能改的文件。

2. 让 Agent 搜索相关代码并给出计划。

3. 让 Agent 做最小 patch。

4. 运行测试或构建。

5. 查看 `git diff`，手动审查结果。

6. 要求 Agent 写一段 PR summary 和验证说明。

  

完成后记录：哪些信息给少了，哪些测试缺失，哪些改动需要人类判断。

  

## [挑战](https://aiweb3.school/zh/handbook/ai/vibe-coding/#%E6%8C%91%E6%88%98)

  

在自己的电脑上安装并配置至少一个 Vibe Coding 工具，例如 Claude Code、Codex CLI 或 OpenCLI，并确认它能在一个测试 repo 中正常使用。

  

完成标准：

  

- 能启动工具并读取当前项目。

- 已配置必要的模型、API key、浏览器桥接或本地工具权限。

- 能让工具完成一个只读任务，例如解释目录结构、搜索相关文件或总结某段代码。

- 能让工具完成一个低风险小改动，例如补一段注释、生成一个脚本或修改一处测试。

- 保留验证记录：工具版本、配置方式、执行命令、关键输出或截图。

  
  

# 模型上下文协议（MCP）

一个让 AI 模型和外部工具/数据源“标准化连接”的协议。

### MCP 不是让模型更聪明，而是让模型“接工具”的方式统一

  

现在每个 AI 应用要接文件、数据库、API、浏览器等，都是自己写一套代码。MCP 想做的是 **AI 领域的 USB-C 接口标准**：

  

- 你有一个工具（比如读本地文件、查 GitHub、调合约），只要实现 MCP Server 端

- 任何 AI 客户端（IDE、聊天应用、Agent 运行时）只要支持 MCP Client，就能用这个工具

- 不需要每个工具为每个客户端重新适配

  

> 所以 MCP 解决的是“生态碎片化”问题，不是模型能力问题。

  

AI 要真正干活，不能只靠 prompt 里的文字。它需要读文件、查数据库、调 API、操作工具。但如果没有统一标准，每个应用都自己发明一套“工具调用方式”，那么：

  

- 同一个工具要为不同客户端写很多遍适配代码

- 权限、参数格式、错误处理、日志都很难统一

- Agent 生态会非常碎

  

MCP 把这层连接**抽象成协议**。你可以把它理解成：**AI 应用的“工具接口标准”**——客户端和 Server 只要都遵守这个协议，就能互相协作。

  
  

|原则|大白话|

|---|---|

|模型不应该直接拥有世界|模型不是“什么都能干”，而是通过明确协议去访问**被授权**的上下文和工具。|

|工具要有 schema|没有参数格式描述，模型调用工具就会变成“自然语言猜参数”，很容易错。|

|权限要在协议外也成立|MCP 可以描述工具需要什么权限，但真正的授权、审计、隔离**还要由你的系统自己实现**。|

|错误要可传递|工具失败、超时、权限不够，必须明确返回错误信息，不能让模型去“猜”发生了什么。|

### Server（服务端）

MCP Server 是 **提供能力的那一侧**。它把工具、资源、提示词等暴露出来，让 AI 客户端可以读取或调用。

  

**Server 的设计重点（边界要清楚）：**

  

- 暴露哪些资源（比如只暴露 `/docs` 目录，不暴露整个硬盘）

- 哪些工具是只读的，哪些会有副作用（写文件、发请求）

- 参数格式是否清楚

- 错误如何返回

- 是否需要用户授权

- 日志和审计在哪里记录

  

> 一个不加限制的 MCP Server，很容易把本地文件、内部 API 或高风险动作暴露给模型——相当于开了个后门。

  

### Client（客户端）

MCP Client 是 **连接模型和 MCP Server 的那一侧**，比如桌面应用、IDE、Agent 运行时、聊天客户端。

  

**Client 的职责：**

  

- 发现 Server 暴露了哪些能力

- 把工具信息告诉模型（通过 prompt 或 function calling）

- 把模型生成的调用请求转成 MCP 协议消息发给 Server

- 负责用户确认、权限提示、会话隔离、展示工具调用过程

  

> 判断一个 Client 是否靠谱：看它是否让用户清楚地知道——当前连接了哪些 Server，模型能调用哪些工具，调用前需不需要确认。

  

---

  

### Tool Schema（工具参数描述）

Tool Schema 就是一份 **工具的“说明书”**，告诉模型：

  

- 这个工具叫什么名字

- 什么时候该用（什么场景）

- 参数是什么意思（比如 `file_path` 是要操作的文件路径）

- 哪些参数必填

- 会不会修改外部状态（只读还是写入）

- 失败时返回什么格式

  

> Schema 写得模糊，模型就会用错误参数去填空，导致工具调用失败或出错。

  

**关联：** 工具描述本质上也是 prompt 的一部分；Agent 的工具调用能力要和状态、权限、停止条件一起设计。

  

---

  

### Permission（权限）

  

MCP 让工具连接更方便，但**方便不等于安全**。读一个文档、查一个 issue、创建一个 PR、发一笔支付、删除一个文件——风险完全不同。

  

**权限至少要区分：**

  

|维度|说明|

|---|---|

|只读 vs 写入|读文档可以自动，写文件必须确认|

|当前会话授权 vs 长期授权|这次同意，下次要不要再问？|

|是否需要用户确认|高风险动作弹窗确认|

|是否访问敏感信息|比如私钥、token、个人数据|

|是否产生外部副作用|发邮件、转账、改数据库|

|是否可撤销和审计|能查谁、什么时候、调了什么工具|

  

> MCP Server 暴露能力之前，应该先定义好权限模型，而不是等到出问题再补。

  

---

  

## 在 AI x Web3 中的位置

  

MCP 在 AI x Web3 里可以作为 **Agent 连接链上工具和开发工具的接口层**。比如：

  

- 读取区块浏览器（查询交易、地址余额）

- 查询合约文档和 ABI

- 调用 RPC 节点

- 生成交易草稿

- 读取项目的 GitHub issue

  

**但 MCP 本身不是钱包安全方案。** 它可以标准化工具的调用格式，但不能替代：

  

- 账户权限

- 交易模拟

- 签名确认（私钥不在 MCP 里）

- session key 和审计日志

  

> 比较稳的设计：**MCP 负责工具发现和调用格式**；**Web3 账户系统负责最终权限和执行边界**。两者各管各的。

  

## [最小实践](https://aiweb3.school/zh/handbook/ai/mcp/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

  

做一个只读 MCP server。

  

它只暴露两个工具：

  

- `search_docs(query)`：搜索本地项目文档。

- `get_file(path)`：读取白名单目录里的文件。

  

要求：

  

- 不能读取白名单外路径。

- 返回结果必须带来源路径。

- 工具调用要写日志。

- 错误要明确返回，而不是静默失败。

  

完成后，再设计一个“写入工具”的权限升级方案：什么时候需要用户确认，如何撤销授权，如何审计每次调用。

  

# 评估（Evaluation）

如何从“凭感觉觉得效果不错”变成“能持续、可靠地判断系统好坏”。

  

## 评估不是写报告，而是让团队能回答“这次改动到底变好还是变坏了”

AI 应用最大的痛点是**输出不稳定**。你改了一句 prompt，可能 A 类问题变好了，B 类问题却变差了；你换了个模型，平均分高了，但某个关键场景反而错了；你接了 RAG，答案变长了，但引用反而更不准确。

  

评估要解决的就是：用**明确的样本、指标、评分方式、回归测试**来判断系统是不是真的进步了。目标不是做一份漂亮报告，而是让团队能回答：**这次改动有没有让关键任务更可靠？有没有引入新的失败模式？**

  

---

  

## 第一性原理（核心观点）

  

|原则|大白话|

|---|---|

|不能被重复测量的 AI 行为，就不能被稳定改进|你没法改进一个“说不清好不好”的东西。|

|先测任务，不只测模型|用户关心的是“整件事能不能完成”，不是模型在某个榜单上的分数。|

|先保住关键失败场景|高风险错误、常见问题、边界条件，要先放进回归测试集，别再犯第二次。|

|评估要贴近产品|离真实用户输入越远，评估越容易变成“自欺欺人”。|

  

---

  

## 知识节点（五个核心概念）

  

### 1. Harness（评估运行框架）

  

**大白话：**

Harness 就是**跑 eval 的“测试架子”**。它负责：

  

- 喂样本（输入问题）

- 调用你的 AI 系统

- 收集输出

- 运行打分（grader）

- 记录结果

  

**一个最小 Harness 需要包含：**

  

- 输入样本

- 期望输出或评分规则

- 被测系统的版本号

- 模型和参数配置

- 运行日志

- 结果报告

  

> Harness 的价值是**可重复**。没有可重复运行的 eval，你就没法比较不同 prompt、不同模型、不同检索策略谁更好。

  

---

  

### 2. Golden Set（黄金样本集）

  

**大白话：**

Golden Set 是一批**精挑细选、人工标注好的测试样本**。它不需要很大，早期 30~100 条高质量样本，往往比一堆随便收集的问题更有用。

  

**Golden Set 应该覆盖：**

  

- 常见正常问题

- 边界问题（模棱两可、信息不全等）

- 容易误判的问题

- 高风险问题（答错了会出事）

- 历史 bug（曾经错过的）

- 用户真实反馈样本

  

> 实用做法：每修一个重要 bug，就把它变成一个 regression 样本加到 Golden Set 里。

  

---

  

### 3. LLM-as-Judge（用模型当评委）

  

**大白话：**

用一个模型（比如 GPT-4）去给另一个模型的输出打分。适用于评估**开放式答案**，比如：摘要质量、是否回答完整、是否遵循格式、是否引用了来源。

  

**但 LLM-as-Judge 不是万能的：** 评委模型也会有偏见、会漏判、会被输出风格影响（比如写“我很确定”可能得分更高，但内容不一定对）。

  

**更稳的做法：**

  

1. 能用规则自动判断的（如是否包含关键词、格式对不对）→ 用规则

2. 开放式质量 → 用 LLM judge

3. 高风险样本 → 保留人工抽检

4. 定期校准 LLM judge 和人工评分的一致性

  

> 记住：LLM-as-Judge 是评估工具，不是最终真相。

  

---

  

### 4. Regression（回归测试）

  

**大白话：**

防止**旧问题复发**。AI 应用很容易出现“修 A 坏 B”——你改了一个地方，之前修好的问题又冒出来了。

  

Regression set 的作用是：把历史问题固定下来，每次改动都重新跑一遍，确保没退步。

  

**一个实用流程：**

  

1. 用户反馈一个错误

2. 你复现并记录输入

3. 标注期望输出（或拒答条件）

4. 加入 regression set

5. 之后每次发布前，必须跑通这个 set

  

> 没有 regression，你就会反复踩同一个坑。

  

---

  

### 5. Observability（可观测性）

  

**大白话：**

Eval 多发生在**发布前**，Observability 发生在**线上真实使用中**。它是让你能在线上看到系统实际表现的能力。

  

**至少应该记录：**

  

- 输入类型和来源（谁、在什么场景下问的）

- 检索结果（RAG 拿到了哪些 chunk）

- 工具调用（调了什么工具，参数是什么，结果如何）

- 模型输出

- 错误和重试

- 用户反馈（赞、踩、纠错）

- 成本和延迟

  

> 没有 observability，你就不知道真实用户在哪里失败，也不知道该往 golden set 里补什么样本。

  

---

  

## 在 AI x Web3 中的位置

  

**大白话：**

Web3 系统里评估**更重要**，因为错误可能直接影响资产、权限、治理判断和链上执行。

  

**需要特别评估的点：**

  

- 交易解释是否准确（别把转账解释成授权）

- 风险提示是否漏报（有漏洞的合约没警告）

- 工具调用参数是否越界（比如让 Agent 调了不该调的 admin 函数）

- 是否能拒绝不确定请求（信息不足时说不确定）

- 是否能识别 Prompt Injection（恶意输入骗过系统）

- 引用和来源是否可追溯（每一句话能点回原文）

- 高风险动作是否要求 human check

  

> Eval 不会替代交易模拟和权限控制，但它能让你**持续发现系统在什么场景下不可靠**，然后针对性改进。

  
  

## [最小实践](https://aiweb3.school/zh/handbook/ai/evaluation/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

  

给一个“交易解释 / 文档问答 / Agent 工具调用”原型做最小 eval。

  

准备 30 条样本：

  

- 10 条正常问题

- 10 条边界或容易混淆的问题

- 5 条历史 bug 或预期失败样本

- 5 条恶意或注入样本

  

为每条样本定义：

  

- 输入

- 期望行为

- 必须包含的信息

- 必须拒绝或提醒的情况

- 是否需要引用来源

  

然后每次改 prompt、模型或检索策略前后跑一遍，记录变化。

  
  

# 微调（Fine-tuning）

很多人一遇到模型效果不好就想到“微调一下”，但实际工程里它往往不是第一步。

  
  

## 微调不是“模型效果不好就训练一下”

  

**人话解释：**

遇到模型回答不好，**先排查别的，最后再想微调**：

  

1. 是不是 **prompt 没写清楚**？

2. 是不是 **上下文缺了关键信息**（RAG 没找到）？

3. 是不是 **输出格式没有 schema**（导致模型乱填）？

4. 是不是 **模型本身能力不够**（比如让 GPT-3.5 做复杂推理）？

5. 有没有 **eval 数据证明**这个问题是**稳定存在**的，而不是偶发？

  

微调的价值在于：让模型在**一类任务上更一致**（比如固定输出格式、特定风格、领域术语）。

它**不适合**用来补实时知识、修权限边界、替代评估。

  

> 没有数据、没有评估、没有明确目标的微调，只会把问题变得更难排查。

  

---

  

## 第一性原理（核心观点）

  

|原则|大白话|

|---|---|

|微调改的是模型行为分布，不是产品系统边界|微调能让模型“更习惯你的数据”，但不会自动带来事实正确、权限安全、引用可靠。|

|先有 eval，再谈微调|没有评估，你不知道微调后是真好，还是只在几个样本上变顺了。|

|先修数据，再修模型|坏数据会把坏习惯训练得更稳定。数据脏，微调后更脏。|

|别用微调存实时知识|实时价格、文档更新、用户数据，放在检索（RAG）或工具里，不要硬训练进模型。|

  

---

  

## 知识节点（五个核心概念）

  

### 1. SFT（监督微调）

  

**大白话：**

给模型一堆 `(输入, 期望输出)` 的例子，让它学着模仿这种“问-答”方式。

  

**适合什么：**

  

- 固定格式输出（比如 JSON 格式的风险报告）

- 特定语气或风格（比如严谨的审计注释）

- 特定任务流程（比如先解释再给建议）

- 领域术语和回答习惯

- 工具调用的样式（比如参数怎么填）

  

**注意：** SFT 对数据质量非常敏感。样本里有错误、格式不一致、边界不清，模型会把它们全学进去，而且学得更稳定。

  

---

  

### 2. LoRA（低秩适配）

  

**大白话：**

一种**省钱省显存**的微调方法。不直接更新整个大模型，而是训练一小部分额外的参数（adapter），然后插到原模型上使用。

  

**适合：**

  

- 开源模型微调

- 资源有限（显存不够）

- 想快速试验不同任务，每个任务存一个小 adapter 文件

  

> LoRA 不是魔法。任务定义、数据质量、评估方法仍然决定最终效果。

  

---

  

### 3. PEFT（参数高效微调）

  

**大白话：**

LoRA 是 PEFT 的一种。PEFT 是一类方法的总称，都干同一件事：**用很少的可训练参数，让大模型适配新任务**。

  

**适合 PEFT 的场景：**

  

- 模型很大，全量微调太贵

- 任务范围明确

- 数据量中等（几百到几千条）

- 需要同时维护多个版本 adapter，换任务就换 adapter

  

---

  

### 4. Dataset（数据集）

  

**大白话：**

微调的核心资产不是“很多问答对”，而是**有清晰定义、质量可控的数据集**。

  

**至少要区分：**

  

- **训练集**：用来训练

- **验证集**：用来调参、选版本

- **测试集**：最终评估（**绝对不能**在训练时看）

- **回归集**：防止修好的问题又复发

  

> 不要把测试集拿去训练，否则评估就失去意义了。

  

---

  

### 5. Overfitting（过拟合）

  

**大白话：**

模型把训练数据**背得太死**，换一个新问题就崩了。

  

**微调时容易过拟合的原因：**

  

- 训练样本太少（几十条）

- 样本风格太单一（全是同一类问法）

- 训练轮数太多

- 目标格式过度僵化（比如每个输出必须 50 个字）

- 验证集和训练集太像（没测出泛化能力）

  

> 过拟合的结果：在准备的例子上看起来 100 分，真实用户一问就露馅。

  

---

  

## 在 AI x Web3 中的位置

  

**大白话：**

微调在 Web3 场景里**可以**用于特定任务，比如：

  

- 交易解释的风格（统一输出格式）

- 治理摘要的格式（提案→争议点→风险）

- 风险标签的输出（固定几类标签）

- 合约注释的风格（统一术语）

- 工具调用的样式（参数怎么写）

  

**但它不适合替代：**

  

- 链上实时状态查询（余额、价格、提案状态）

- 合约安全审计（审计要专用工具）

- 交易模拟（模拟要真实执行）

- 钱包权限控制（签名不能靠模型）

- 用户确认（高风险必须人点头）

- 外部事实引用（最新文档靠 RAG）

  

> 一句话：微调可以让模型**更懂你的格式和习惯**，但不能让模型直接变成**可信执行层**。

  
  
  
  
  
  
  

## [最小实践](https://aiweb3.school/zh/handbook/ai/fine-tuning/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

  

做一个“结构化摘要格式”的微调前评估。

  

先不要训练。先准备 50 条样本：

  

- 输入：一段技术文档或提案。

- 输出：固定 JSON 字段，例如 `summary`、`risks`、`open_questions`、`sources`。

  

然后比较三种方案：

  

1. 只改 prompt。

2. Prompt + few-shot。

3. 小规模 fine-tuning 或 adapter 方案。

  

每种方案都用同一套 eval 检查：

  

- 字段是否完整

- 是否编造来源

- 风险点是否漏掉

- 输出是否稳定

  

只有当前两种方案无法稳定解决问题时，再考虑 fine-tuning。

  
  

# 推理服务（Inference）

模型训练好之后，真正在产品里**跑起来、接请求、给回答**的那一层。

  

## 推理不是“一个云服务按钮”，而是延迟、成本、隐私、稳定性的综合选择

  

很多人做个 demo 时，直接调 API 就行了。但产品一旦有**真实用户、真实流量、真实任务**，问题就来了：

  

- 响应太慢怎么办？（用户等不了 5 秒）

- 成本怎么控制？（一天调几万次，账单爆炸）

- 模型升级了，输出格式变了怎么办？

- 用户隐私数据能不能送到第三方 API？（合规问题）

- 本地跑模型够不够用？

- 服务崩了怎么降级？

  

推理这一层，就是决定**模型能力如何被产品稳定、可控、可观测地消费**。没有好的推理服务，再强的模型也只会让线上问题更难排查。

  

---

  

## 第一性原理（核心观点）

  

|原则|大白话|

|---|---|

|推理服务的核心是“在约束条件下交付可用答案”|不是只要能跑出答案就行，而是要**快、不贵、稳定、安全**。|

|同一个模型，不同部署方式表现完全不同|API 模型、本地 GPU、量化后的小模型、边缘设备——延迟、成本、质量都不一样。你选的不只是模型，是一套**权衡**。|

|质量有代价|更强模型 = 更高成本 / 更长延迟 / 更复杂部署。|

|部署改变边界|API 省心但送数据出去；本地可控但自己要管硬件。|

|服务要可替换|把模型调用封装好，才能做 fallback、灰度、评估——别写死在代码里。|

  

---

  

## 知识节点（四个核心概念）

  

### 1. API Model（云服务模型）

  

**大白话：**

就是常见的调用 OpenAI、Anthropic、国内厂商的 API。输入请求，收到回答。

  

**优点：** 上手快、模型更新快、不用管基础设施。

**缺点（容易被忽略的工程问题）：** 你仍然要处理——速率限制、超时、重试、日志脱敏（别把用户密码记到日志里）、账单控制、模型版本变更导致输出退化。

  

> 尤其在 Agent 场景：用户问一个问题，背后可能触发 5 次模型调用，成本和延迟会被放大。

  

---

  

### 2. Local Model（本地模型）

  

**大白话：**

在自己服务器、自己电脑或私有环境里跑模型。适合：**隐私要求高、成本敏感、需要离线运行、需要深度定制**。

  

**限制也很直接：**

  

- 模型多大？显存够不够？

- 并发能力？单卡只能串行处理多个请求？

- 上下文窗口？

- 推理框架（vLLM、TGI、llama.cpp）调优？

- 量化后质量掉多少？

  

> 对大多数团队来说，本地模型不是替代所有 API，而是承担**特定任务**：分类、抽取、代码补全、轻量 Agent、隐私数据处理，或者作为 API 的 fallback。

  

---

  

### 3. Quantization（量化）

  

**大白话：**

把模型的计算精度降低（比如从 FP16 降到 INT8、INT4），让模型**更小、更快、更省显存**，但可能**牺牲一点质量**。

  

**为什么需要：** 让普通电脑、小 GPU、甚至手机能跑大模型。

  

**问题：** 量化可能降低质量，尤其在**长推理、代码生成、多语言、数学、工具调用**等任务上。判断量化模型是否可用，**不能只靠聊天感觉**，要用你自己的任务样本去测。

  

> 量化的取舍：体积↓、速度↑、质量？→ 必须实测。

  

---

  

### 4. Serving（模型服务化）

  

**大白话：**

把模型变成一个**稳定的、可供应用调用的服务**。不只是“跑起来”，还要考虑：

  

- 并发（多个请求同时来怎么办？）

- 队列（忙不过来时排队还是拒绝？）

- 流式输出（一个字一个字蹦出来，用户体验）

- 缓存（相同输入直接返回上次结果）

- 监控（延迟、错误率、成本）

- 失败处理（重试？降级到小模型？）

- 版本管理（灰度上新模型，出问题立马回滚）

  

> 一个成熟的推理服务，至少要能回答：请求失败怎么重试？模型版本怎么灰度？输入输出日志如何脱敏？长请求进不进队列？

  

**Serving 做不好，模型越强，线上问题越难排查。**

  

---

  

## 在 AI x Web3 中的位置

  

**大白话：**

Web3 项目经常要：读链上数据 → 送进模型 → 给出交易解释、合约分析、策略建议或 Agent 执行。推理这一层直接影响：

  

- 用户**能不能安全地等待结果**（延迟太长，用户可能不耐烦就瞎签了）

- 是否**愿意支付成本**（每次分析花几分钱，用户可能接受）

- Agent 是否**能在合理时间内完成链上前置判断**（比如模拟交易）

  

**特别要求：** 如果模型要参与**交易解释、合约分析、策略建议或 Agent 执行**，推理服务必须留下**可审计记录**：

  

- 用了哪个模型（版本、参数）

- 输入来自哪里（哪个用户、哪条链上数据）

- 输出是什么

- 是否触发工具调用（比如调了 RPC）

- 失败时怎么处理的

  

> 链上动作不可逆，推理层就不能像普通聊天一样随意。审计是必须的。

  

## [最小实践](https://aiweb3.school/zh/handbook/ai/inference/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

  

做一个最小推理对比：

  

1. 选择同一个任务，例如“总结一笔交易的风险”或“从合约 ABI 中提取可调用方法”。

2. 用一个托管 API 模型跑一次。

3. 用一个本地模型或较小模型跑一次。

4. 记录延迟、成本、输出质量、隐私边界和失败情况。

5. 写出你会在产品中选择哪一种，以及 fallback 怎么设计。

  

重点不是证明哪个模型最好，而是学会把模型选择放进真实约束里。
